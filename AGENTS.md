## What This Project Is

`@cyguin/sniplet` is a drop-in snippet sharing npm package for Next.js apps.
It follows the `next-auth` / `uploadthing` pattern: install, wire two files, feature works.
No external service. No separate deployment. Developer owns their data.

Package name: `@cyguin/sniplet`
npm scope: `@cyguin`
Repo: `cyguin/sniplet`
Docs URL: `sniplet.cyguin.com`
License: MIT

## Critical Implementation Constraints

### Burn-on-Read — MUST be atomic

Never implement burn-on-read as SELECT then UPDATE. This creates a race condition.

**Postgres — use UPDATE ... RETURNING:**
```sql
UPDATE sniplet_snips
SET burned_at = NOW()
WHERE id = $1
  AND burn_on_read = TRUE
  AND burned_at IS NULL
RETURNING *;
```
If RETURNING result is empty → snip was already burned → return 410 Gone.

**SQLite — use better-sqlite3 transaction:**
```typescript
const tx = db.transaction((id: string, now: string) => {
  const result = db.prepare(
    `UPDATE sniplet_snips
     SET burned_at = ?
     WHERE id = ? AND burn_on_read = 1 AND burned_at IS NULL`
  ).run(now, id)
  return result.changes > 0
})
const wasBurned = tx(id, new Date().toISOString())
if (!wasBurned) throw new SnipAlreadyBurnedError(id)
```

### Database Schema — use this exactly, namespaced to avoid host app collision

```sql
CREATE TABLE IF NOT EXISTS sniplet_snips (
  id            TEXT PRIMARY KEY,
  content       TEXT NOT NULL,
  language      TEXT,
  expires_at    TIMESTAMPTZ,
  burn_on_read  BOOLEAN DEFAULT FALSE,
  burned_at     TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS sniplet_expires_idx
  ON sniplet_snips(expires_at)
  WHERE expires_at IS NOT NULL;
```

Migration must run idempotently on first handler call. Use `CREATE TABLE IF NOT EXISTS`
and `CREATE INDEX IF NOT EXISTS`. Never use a migration library — raw SQL only.

### ID Generation

Use `nanoid` with default alphabet and length 12. Never use uuid. Never use Math.random().

```typescript
import { nanoid } from 'nanoid'
const id = nanoid(12)
```

### Error Handling Convention (critical for this project)

All errors are typed. Define a base `SnipletError` class and specific subclasses:

```typescript
export class SnipletError extends Error {
  constructor(message: string, public readonly code: string) {
    super(message)
    this.name = 'SnipletError'
  }
}

export class SnipNotFoundError extends SnipletError {
  constructor(id: string) {
    super(`Snip not found: ${id}`, 'SNIP_NOT_FOUND')
  }
}

export class SnipAlreadyBurnedError extends SnipletError {
  constructor(id: string) {
    super(`Snip already burned: ${id}`, 'SNIP_ALREADY_BURNED')
  }
}

export class SnipExpiredError extends SnipletError {
  constructor(id: string) {
    super(`Snip expired: ${id}`, 'SNIP_EXPIRED')
  }
}
```

The route handler maps these error classes to HTTP status codes:
- `SnipNotFoundError` → 404
- `SnipAlreadyBurnedError` → 410
- `SnipExpiredError` → 410
- Unknown error → 500, log to stderr, never leak internals to response

Every function in `core/` throws on error. Every function in `next/` catches and maps to
HTTP. Never mix throw and return-error in the same layer.

### Expiry Check Order

When a snip is retrieved, check in this order:
1. Does the snip exist? If not → SnipNotFoundError
2. Is it burned? If `burnedAt !== null` → SnipAlreadyBurnedError
3. Is it expired? If `expiresAt !== null && expiresAt < new Date()` → SnipExpiredError
4. Is it burn-on-read? If yes → atomic burn, then return content
5. Return content
